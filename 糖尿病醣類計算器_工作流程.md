## 一、產品定位

一個「先計算、再紀錄、後學習」的糖尿病飲食工具。重點不是只算碳水，而是讓每次用餐的劑量決策有歷史依據——同一道菜累積愈多次紀錄，下次劑量建議愈準。

**使用裝置：胰島素幫浦**（最小劑量單位 0.1U），所有劑量輸入/輸出/演算法皆以 0.1U 精度設計。注射筆使用者本工具不適用（精度不同）。

不是醫療裝置，不取代醫師。所有建議只是統計輔助。

## 一之二、劑量精度規則（重要）

| 項目 | 規則 |
|---|---|
| 劑量輸入欄位 | step="0.1"，數字鍵盤含小數點 |
| 劑量顯示格式 | `X.X U`（永遠小數一位，例 4.2U、5.5U、6.8U；整數也寫 `6.0 U`） |
| I:C ratio 計算 | round 到 0.1（例 1:9.4） |
| 「最佳劑量建議」步進 | 0.1U |
| 餐前校正劑量（v2 ISF） | 0.1U |
| 圖表 y 軸標籤 | 顯示到小數一位 |

## 二、兩大功能

### 功能 A — 單餐 carb + GI 計算器

| 項目 | 說明 |
|---|---|
| 食材選擇 | 從食材庫挑 → 輸入克數，可加多個 |
| 即時計算 | 總碳水（g）、淨碳水（g，扣纖維）、加權 GI、加權 GL |
| 視覺化 | 每食材碳水佔比（圓餅或長條） |
| 自訂食材 | 手動輸入碳水/纖維/GI，存進個人庫 |
| 常吃餐 | 不預設便當組合——使用者用「常吃餐記憶」自建（功能 B） |

**加權 GI 公式**

```
weighted_GI = Σ(食材_GI × 食材_可吸收碳水) / Σ(食材_可吸收碳水)
GL = (weighted_GI × 總可吸收碳水) / 100
```

「可吸收碳水」= 總碳水 − 纖維（淨碳水）。沒纖維資料的食材以總碳水代入。

### 功能 B — 用餐＋胰島素＋CGM 日誌 → 推算最佳劑量

#### 每筆用餐紀錄欄位

- 日期時間
- 餐別（早 / 午 / 晚 / 宵夜）
- 食材清單（從 A 帶過來，含克數、各別碳水）
- 總碳水 / 淨碳水 / 加權 GI / GL
- **胰島素劑量（U，0.1U 精度）**
- **Bolus 類型**（純 / square / dual，v1 預設純）
- **延伸時長**（分鐘，僅 square/dual 顯示，v2）
- 餐前 BG（mg/dL）
- 餐後 1h BG
- 餐後 2h BG
- 備註（運動、生病、月經、酒精 等情境標籤）
- **照片**（可選）：
  - `photo_url`：Cloudinary URL（壓縮後 800px JPEG）
  - `photo_phash`：64-bit pHash 字串（用於相似度比對，不需重新下載照片）
  - `photo_thumb_b64`：縮圖 base64（200px，~10KB，內嵌紀錄供列表快顯）
  - `upload_pending`：boolean，true 表示主照片還沒成功上傳（暫存 IndexedDB）

#### I:C ratio 計算

每筆紀錄即時算：
```
I:C = 該餐碳水(g) ÷ 該餐胰島素(U)
```

平均/趨勢按餐別分群（早 / 午 / 晚 / 宵夜），因為很多成人早餐 ratio 不同（晨間胰島素抗性）。

#### 「食譜記憶」功能

- 每筆紀錄可選「儲存為常吃餐」+ 命名（例：「全聯雞腿便當」、「家裡早餐燕麥」）
- 同道菜累積 N 次後，下次選同道菜時帶出歷史 + 建議劑量
- 同一道常吃餐 = 一個 series（共用名稱 + 累積紀錄）
- 自動歸類：新紀錄拍照後，系統用照片相似度（pHash）跟過去 30 天紀錄比對，高度相似就建議歸到同一 series（詳見下方）

#### 照片相似度自動歸類

**重要：本工具不做食物 AI 辨識、不估克數、不估碳水。**只用「圖片相似度比對」幫使用者把同一道菜的多次紀錄歸類在一起。

**v1 採用：Perceptual Hash (pHash)**

- 純 JavaScript 實作（client-side、無 API、零成本、隱私本地）
- 從照片提取 64-bit hash（依色彩分布、邊緣、佈局）
- 比對新照片跟過往紀錄的 hash → 計算 Hamming distance
- 距離 < threshold（預設 8）→ 跳出建議「這跟 X 月 Y 日的『鮭魚握壽司套餐』相似，要不要歸成同一筆？」

**互動流程**

1. 新增紀錄 → 拍照（或從相簿選）
2. 系統自動跑 pHash，與過去 30 天紀錄的 pHash 比對（IndexedDB 查詢）
3. 找到 distance < threshold 的候選 → 列出 top 3 相似紀錄（顯示縮圖、名稱、日期、相似度）
4. 使用者：
   - 「歸到這一筆」→ 自動帶入該 series 的名稱、預設食材、上次劑量；繼續編輯
   - 「都不是 / 新增」→ 創新 series，輸入餐名
5. 沒有相似紀錄 → 不彈窗，正常編輯

**門檻 threshold（Hamming distance）**

| 值 | 行為 |
|---|---|
| ≤ 5 | 「幾乎一定是同一道」 |
| 6–8 | 「很可能是同一道」（預設彈窗門檻） |
| 9–12 | 「可能相關」（次佳推薦，列入 top 3） |
| ≥ 13 | 不顯示 |

設定頁可調整門檻（5–15），讓使用者依自己拍照習慣校準（光線變化大的人可調寬鬆）。

**儲存（Cloudinary 雲端版，v1）**

- 每張照片：壓縮（最長邊 800px、JPEG quality 0.7）→ unsigned upload 到使用者自己的 Cloudinary
- Cloudinary 回傳的 URL（含隨機 hash，難以猜測）存進紀錄的 `photo_url` 欄位
- pHash（64-bit 字串）+ 縮圖（base64，最長邊 200px，~10KB）一併存在紀錄 JSON 內，連同 photo_url 同步到 Gist
- 顯示主圖時：直接從 Cloudinary URL 載（瀏覽器有快取）
- 切換裝置：從 Gist pull 紀錄 → 縮圖立刻顯示、主圖按需從 Cloudinary 載
- pHash 比對：完全用 Gist 裡的 phash 字串本地比 Hamming distance，**不需要重新下載任何照片**

**離線/上傳失敗 fallback**

- 拍照壓縮後嘗試上傳 → 成功則 photo_url 存入紀錄
- 失敗（網路不通、Cloudinary 暫時故障）→ 暫存到 IndexedDB（`pendingUploads` store）+ 紀錄標 `upload_pending: true`
- 下次 app 開啟有網路 → 自動重試 pending 上傳；成功後寫回 photo_url、清掉 pending、更新 Gist
- IndexedDB 只當待重傳暫存用，**不**做為主要照片存放

**Cloudinary 設定**

- 使用者在設定頁輸入 Cloud Name + unsigned Upload Preset（一次性，存 localStorage）
- 使用 unsigned preset：上傳不需 API key（公開的 cloud name 沒風險，只能上傳、不能刪/管理）
- Cloudinary 免費版 25GB，個人使用足夠
- 不做：signed URL（v2 候選）、雲端照片管理（刪/移動）、多帳號切換

**未來升級路徑（v2 候選）**

- 升級到 TensorFlow.js + MobileNet（~10MB 模型，cosine similarity）：對「同道菜不同角度／光線」更穩健
- 觸發條件：使用者覺得 pHash 漏判太多 → 在設定頁開「進階比對」開關
- Hybrid 流程：先 pHash 快比；沒中再按「找相似」用 MobileNet 二次比對

#### 自動建議餐名（系統提議，使用者可改）

**核心原則**：系統主動建議餐名，但使用者隨時可改。不破壞命名自由。

**三層 fallback 邏輯**（優先順序）

| 層級 | 條件 | 建議名 |
|---|---|---|
| 1. 照片匹配 | pHash distance < 8 與某筆紀錄相似 | 帶出該筆紀錄的餐名 |
| 2. 食材匹配 | 食材清單相同、份量誤差 < 10% 與某筆紀錄相符 | 帶出該筆紀錄的餐名 |
| 3. 自動 compose | 都沒匹配 | 取碳水佔比最高的前 3 個食材：「白米飯 + 雞胸肉 + 花椰菜」 |
| 3b. 退路 | 沒選食材但有時段 | 「早餐 0501」「午餐 0501」「晚餐 0501」「宵夜 0501」 |

預設用層級 3 的食材組合命名，因為一看就懂；只有完全沒食材時才退到時段名。

**食材匹配（層級 2）細節**

兩筆紀錄食材清單視為「相同」需滿足：
- 食材集合完全一致（同 IDs）
- 每個食材克數差 < 10%（例：白飯 150g vs 145g 通過；150g vs 130g 不通過）
- 任一條件不滿足 → 不帶用既有名

**自動 compose（層級 3）規則**

1. 計算每個食材的碳水貢獻（= 該食材碳水 g）
2. 排序由高至低，取前 3
3. 用「+」連接：例 `白米飯 + 雞胸肉 + 花椰菜`
4. 若只有 1-2 個食材碳水 > 0，就只列那 1-2 個
5. 全部 0 碳水（極少見的全肉餐）→ 退到時段名

**UI 行為**

- 餐名欄是 input，預設帶建議名（淡灰底文字 + 小字提示「建議名，可改」）
- 使用者點 input → 變正常編輯狀態，可改成個人化名字（「鮭魚握壽司套餐」、「老媽滷肉飯」、「我家便當」）
- 儲存時：使用者編輯過 → 用編輯後；沒編輯 → 用建議名
- 同名次紀錄自動歸到同一個 series（v1 嚴格 string 比對；v2 加 fuzzy）

**邊界**

- 使用者改名後想「合併餐記錄」（把過去散落的紀錄歸到新名字底下）→ v2 提供「合併」功能；v1 暫不做
- 兩個 series 若使用者命名相同 → v1 視為同一個 series（合併）；如有衝突在儲存時提示

#### 餐名建議與相似照片建議的互動順序

新增紀錄流程：

1. 拍照（可選）→ 跑 pHash
2. 若照片匹配 → 跳「相似建議 modal」，帶出名字 → 使用者選歸類 → 餐名 input 預填該名
3. 若照片沒匹配但食材匹配 → 餐名 input 預填食材匹配到的那筆名字
4. 若都沒匹配 → 餐名 input 預填自動 compose 的「白米飯 + 雞胸 + 花椰菜」
5. 任何階段使用者都可改餐名

#### 「最佳劑量」演算法（v1 草案，幫浦版 0.1U 精度）

對某道菜的歷史 N 筆紀錄，看餐後 1h+2h 平均峰值。

**調整方向**（依平均峰值）

| 條件 | 方向 |
|---|---|
| 平均峰值 < 70 mg/dL | 減量 |
| 平均峰值 70–140 | 維持 |
| 平均峰值 140–180 | 加量（小） |
| 平均峰值 > 180 | 加量（中） |
| 餐前 BG > 200 | 額外校正劑量另算（v2 再做 ISF） |

**調整幅度**（依樣本數，幫浦使用者可微調）

| 樣本 N | 微調幅度（基本） | 大幅 |
|---|---|---|
| N<3 | ±0.1 U | 不建議大幅調 |
| N=3–6 | ±0.2–0.3 U | — |
| N≥7 | ±0.5 U | 信心高才敢調大 |

**舉例**：某道菜 4 次紀錄平均餐後 2h = 195（在 140-180 上、180-200 中段），N=4 → 加 0.2-0.3U。原本 6.0U → 建議 6.2-6.3U。

**信心顯示**：N<3 顯示「樣本不足，僅供參考」；N≥3 顯示「N 次紀錄」；N≥7 顯示「建議信心：高」。

公式僅做平均值版本；中位數 / 標準差版本待 Phase 5 再評估。

圖表：
1. 最近 7/30 天 I:C 變化（折線，按餐別三色）
2. 餐後 BG 散佈（X = 碳水量、Y = 餐後 2h BG，色點 = 劑量）

#### 建議 dose 提示

進入新紀錄時，若選的食材組合與某道「常吃餐」相符（含模糊比對：食材重疊 ≥ 70%），顯示：
> 上次「全聯雞腿便當」用 1:8（4.0U / 32g 碳水），餐後 2h BG = 195。建議這次試 1:7.4（4.3U）。

劑量永遠顯示小數一位。

#### Bolus 類型（幫浦特殊功能）

| 類型 | 用途 | v1 支援 |
|---|---|---|
| 純 bolus（normal/standard） | 一般餐 | ✅ 必須 |
| Square wave（延伸性 bolus） | 高脂高蛋白餐（披薩、起司、油炸物），延長劑量釋放避免後期高血糖 | v2 |
| Dual wave（混合 bolus） | 一部分立刻、一部分延伸 | v2 |

v1 紀錄欄位先預留「bolus 類型」（預設純 bolus），v2 再做 square/dual 的 duration 輸入欄。
若使用者選 square/dual，紀錄表單動態多出「延伸時長（分鐘）」欄位。

## 三、食材庫

### 資料來源

| 來源 | 用途 | 取得方式 |
|---|---|---|
| TFDA 台灣食品營養成分資料庫 | 碳水、纖維、熱量、蛋白、脂肪 | 公開 API 或下載 CSV，整理成 JSON |
| University of Sydney GI Database | GI 主來源（國際） | 網站查詢，手動整理常見食物 |
| 衛福部 GI 表 | 台灣食物 GI | PDF/網頁，手動整理 |
| Glycemic Index Foundation | 補充 | 網頁查詢 |
| 估計值 | TFDA 沒列 GI 的食材 | 白米 ~73、糙米 ~50、地瓜 ~70、燕麥 ~55 等，標「估計」 |

**找不到 GI 的食材**：標「待確認」，不亂填。計算時可選「忽略 GI」或「以類別預設值」。

### 食材分類（chip 顏色用）

走 4 色循環（黃/橘/紅/暖深棕邊框），不像狗狗計算器那種 13 色。分類本身共 7 類，但配色簡化：

1. 主食（米飯麵類）— 飽和黃 `#FFC107`
2. 蔬菜 — 暖深棕邊框 `#3D2914`，白底
3. 蛋白質（肉魚蛋豆）— 紅 `#E53935`
4. 水果 — 橘 `#FB8C00`
5. 飲料 — 暖深棕邊框，白底
6. 便當組合（pre-set） — 飽和黃實心，紅色字
7. 其他 / 自訂 — 暖深棕邊框，白底

### 預設便當組合 ~~（Phase 1 整理目標 ~15-20 道）~~

**v1 不做**——改由「常吃餐記憶」（功能 B）讓使用者自建。

理由：使用者吃的便當與外食組合差異大，「預設便當組合」常常還是要微調；不如讓使用者第一次吃時拍照、選食材、命名後存成常吃餐，下次自動帶出，更貼近個人實際飲食。

（已移到「未做 / 未來考慮」清單）

### 資料儲存

| 層 | 內容 | 同步 |
|---|---|---|
| `localStorage` | UI 設定、GitHub PAT、Cloudinary cloud name + upload preset、最近 cache | 不同步 |
| `IndexedDB` | 用餐紀錄、自訂食材、常吃餐、**待重傳照片暫存**（pendingUploads） | → Gist |
| GitHub Gist | JSON 全量備份（含 photo_url + pHash + 縮圖；不含原圖） | 主同步點 |
| Cloudinary | 主照片（壓縮 800px JPEG），URL 存在 Gist 紀錄裡 | 由 client 上傳 |

**Gist 同步流程**

1. 啟動：pull 最新 Gist → 與本地比對 → 較新者勝（用 `updatedAt` 欄位）
2. 寫入：debounce 2 秒後 push 全量 JSON
3. 衝突：先用「最後寫入勝」（v1 簡單版），未來再做 merge
4. 離線：本地寫入正常，連線後再同步

**PAT 取得流程**

- 設定頁顯示步驟：GitHub → Settings → Developer settings → PATs → Generate（只勾 `gist` scope）
- 使用者貼 PAT → 存 `localStorage`（基本 obfuscation，不假裝是真加密）
- 第一次成功同步後，建立 Gist（私人）並記住 gist_id
- 設定頁可隨時更換 PAT / 撤銷

**安全**：PAT 不寫進 commit / log / file。每次使用後不留 console。

**Cloudinary 設定流程**

- 設定頁有「☁ 雲端照片儲存（Cloudinary）」區塊
- 使用者輸入 **Cloud Name** + **Upload Preset**（unsigned）→ 存 `localStorage`
- Cloudinary 帳號自己申請（免費 25GB）；preset 在 Cloudinary 後台設成 `Unsigned`
- 「測試上傳」按鈕：丟一張 1×1 透明 PNG 到 Cloudinary 驗證設定
- 設定後狀態顯示「✓ 已連接 Cloudinary」+ 用量（從 Cloudinary admin API 抓，可選）

**使用者預設值（Phase 2 程式內初始化）**

| 欄位 | 預設值 |
|---|---|
| Cloud Name | `dcelpklzi` |
| Upload Preset | `carb-meals`（unsigned） |
| Upload endpoint | `https://api.cloudinary.com/v1_1/dcelpklzi/image/upload` |

第一次開 app 時 localStorage 自動寫入這兩個值，使用者不用手動填；設定頁仍然提供修改/清除/測試上傳按鈕，方便未來換。

> **注意**：這兩個值雖然是 unsigned credentials（公開無風險），但仍**不寫進 git commit 訊息或 public README**——習慣上把所有 credentials（即使非機密）放在 client-side runtime 而不是 repo metadata。
>
> 程式內以常數 `DEFAULT_CLOUDINARY_CLOUD_NAME` / `DEFAULT_CLOUDINARY_UPLOAD_PRESET` 寫在 `app.js` 內（client-side 一定看得到），但 commit 訊息與 README 不提及具體值。

**測試上傳的 endpoint**

```
POST https://api.cloudinary.com/v1_1/{cloud_name}/image/upload
Content-Type: multipart/form-data
form: upload_preset={preset}, file=<binary>
```

成功回 200 + JSON（含 `secure_url`、`public_id`、`width`、`height`）。失敗時的常見錯誤：
- 400 + `Upload preset must be whitelisted for unsigned uploads` → preset 沒設成 unsigned
- 401 / 403 → Cloud Name 拼錯或 preset 不存在
- 413 → 檔案太大（免費版單檔 10MB；妳壓縮過的 800px JPEG 通常 <500KB）

**為什麼用 Cloudinary 而不是塞進 Gist 或 IndexedDB？**

- Gist 對單檔有 1MB 限制，多張 800px JPEG 累積會爆
- IndexedDB 只存本地，換手機就沒了
- Cloudinary 給 URL 跨裝置共用，免費額度足夠個人使用
- unsigned preset 的 Cloud Name 即使外洩也只能被別人上傳照片到妳帳號（最多浪費額度，不能讀/刪）

## 七、開發規則
